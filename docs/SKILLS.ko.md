# re:Harmoniz 스킬 — 언제 무엇을 쓰나

[English](SKILLS.md) · **한국어**

진화 스킬 일곱에 설정 위자드 하나, 각각 명령 하나입니다. 굳이 외울 필요는 없습니다. 다음에 뭘 해야 할지 막힐 때는 [`reharm:pushing`](#reharmpushing--지금-뭐-하지)을 돌리면 스코프를 읽고 다음 수를 짚어 줍니다. 이 문서는 직접 고르고 싶을 때를 위한 것으로, *어떤 상황*에 어떤 스킬을 쓰는지, 그리고 각 스킬이 무엇을 건드리고 무엇은 건드리지 않는지를 정리했습니다.

모든 스킬은 평이한 말(한국어·영어)에도 반응하므로, 명령을 외워 치는 대신 하고 싶은 걸 그냥 말해도 됩니다. 트리거 문구는 스킬마다 함께 적어 뒀습니다.

## 스킬 하나씩

### `reharm:root` — 새 자료 들이기

**언제 쓰나:** 새 주제를 막 시작할 때, 또는 기존 주제에 새로 넣을 게 생겼을 때 — repo URL, 논문이나 글, 의사코드, 예전 노트, 머릿속의 막연한 아이디어까지 무엇이든.

**무슨 일이 일어나나:** 원자료는 손대지 않은 채 `.raw/`에 저장되고, 한 페이지짜리 `sources/` 요약이 생기며, 그 안의 사실 단언들이 원자적 `claims/`가 됩니다 — 전부 `seed · generation 1 · confidence low`로 태어납니다. 여러 개를 한꺼번에 넣으면 각각 별도의 격리된 sub-agent에서 소화되므로, 한 출처의 서술 틀이 다른 출처의 주장으로 새어들지 않습니다.

**무엇은 안 하나:** 아무것도 판정하거나 단단하게 만들지 않습니다 — 전부 1세대에서 출발합니다. 등급 매기기는 reharmonization의 몫입니다.

**그다음엔:** `reharm:reharmonization`으로 압력 시험을 시작하거나, 어떤 새 노드부터 진화시킬지 궁금하면 `reharm:pushing`에게 물어보세요.

**이렇게 말해도 됩니다:** *"reharm root"*, *"시드 투입"*, *"스코프 만들어"* — 아니면 그냥 URL을 붙여넣기.

### `reharm:pushing` — 지금 뭐 하지?

**언제 쓰나:** 한동안 놔뒀던 스코프로 돌아왔을 때, 또는 지금 가장 값진 다음 수가 뭔지 딱 안 떠오를 때. **막히면 일단 이걸 돌리세요.**

**무슨 일이 일어나나:** 상태 전체를 읽습니다 — 성숙도 통계, frontier 점수, 열린 모순, 직전 세션의 stagnation 판정, 막혀 있는 근거, deliverable이 없거나 낡았는지 여부 — 그리고 다음 수 하나(씨앗 투입 / 진화 / 판정 / 스코프 연결 / 실험 설계 / 답변 합성 / 막힌 근거의 딥리서치 승격)를 그 근거와 함께 짚어 줍니다.

**무엇은 안 하나:** 아무것도 바꾸지 않고, 아무것도 실행하지 않습니다. 가리키기만 합니다.

**그다음엔:** 추천받은 걸 돌리면 됩니다.

**이렇게 말해도 됩니다:** *"다음 뭐 해"*, *"상태 봐줘"*, *"뭐부터 할까"*.

### `reharm:reharmonization` — 주장을 한 세대 태우기

**언제 쓰나:** 어떤 주장에 새 의심이나 새 자료가 들어왔을 때, 또는 frontier 노드를 밀어올릴 때가 됐을 때. 프로젝트 이름의 유래가 된 핵심 루프입니다.

**무슨 일이 일어나나:** 한 사이클 전체가 돕니다 — **회고 → 표적(어떤 노드를 태울지 당신이 승인) → 변이(수정, 새 출처 분해, 반대 근거 사냥) → 자연선택(반박자 3인, 각자 하나의 렌즈 — 일관성 · 근거 · 재현성; 2/3 이상 생존해야 함) → 기록(`E####.md` + `index`·`hot` 갱신)**. 살아남은 노드는 세대를 얻고 승급할 수 있습니다(`seed → developing → hardened → evergreen`).

**무엇은 안 하나:** 반박을 못 견딘 것은 승급시키지 않고, 당신이 표적으로 승인하지 않은 노드는 건드리지 않습니다. `hardened` 위로 올라가려면 독립 출처와 실세계 현장 근거가 필요합니다 — 의지만으로는 안 됩니다.

**그다음엔:** 세션이 애매한 찌꺼기를 남겼으면 `reharm:critique`, 다음 표적을 고르려면 `reharm:pushing`.

**이렇게 말해도 됩니다:** *"진화 돌려"*, *"진화 세션"*.

### `reharm:critique` — 어중간한 것들 정리

**언제 쓰나:** 백로그가 쌓였을 때 — 열린 질문, 두 노드에 걸쳐 있는 모순, 신뢰도가 낮은 채 정체된 노드, lint 경고. 또 한 번의 반박이 아니라 사람의 판정이 필요한 찌꺼기들.

**무슨 일이 일어나나:** 그 백로그를 모아 당신을 인터뷰합니다. 테마가 비슷한 항목들은 멀티셀렉트 일괄 triage 한 번으로 묶어(승격 / 보류 / 보관) 처리하고, 4세션 이상 손대지 않은 질문은 노화 묶음으로 한꺼번에 제안합니다. 당신의 판정이 반영됩니다 — 모순은 `## Objections & Limits`로 흡수되고, 질문은 더 날카로워지거나 적재됩니다.

**무엇은 안 하나:** 세대는 절대 올리지 않습니다 — 세대는 당신의 판정이 아니라 반박 생존으로만 얻으니까요 — 그래서 `confidence`·`status`만 조정합니다. 그리고 자동으로 보관하지 않습니다: 노화는 언제나 *제안*이지 소리 없이 처리되는 법이 없습니다.

**그다음엔:** 판정으로 길이 트인 것들을 `reharm:reharmonization`으로 진화시키세요.

**이렇게 말해도 됩니다:** *"모호한 것 정리"*, *"노드 리뷰"*.

### `reharm:modal-interchange` — 두 스코프 잇기

**언제 쓰나:** 연구 스코프가 둘 있고, 한쪽의 미해결 문제를 다른 쪽의 메커니즘이 답할 것 같다는 감이 올 때.

**무슨 일이 일어나나:** 두 스코프를 가볍게 정찰해 교차점을 찾아내고, *현재* 스코프에 `mashups/` 통찰을 발행합니다 — 그 `sources:`는 양쪽 원본 노드를 wikilink로 겁니다. (이름은 음악 용어입니다 — 평행 선법에서 코드 하나를 빌려온다는 뜻.)

**무엇은 안 하나:** 다른 스코프의 노드를 복사하거나 수정하지 않습니다 — *인용*할 뿐이라 단일 진실 공급원이 유지됩니다. 그리고 `borrowed:` 스냅샷을 찍어 두어, 나중에 원본이 바뀌면 조용한 부패 대신 objection으로 드러나게 합니다. 새로 발행된 mashup은 `seed`로 태어나 이후 세션에서 다른 노드처럼 자연선택을 거칩니다.

**그다음엔:** `reharm:reharmonization`으로 새 mashup을 단단하게.

**이렇게 말해도 됩니다:** *"모달 인터체인지"*, *"크로스 매쉬업"* — 두 스코프 이름과 함께.

### `reharm:experiment-design` — 실세계 증명 설계

**언제 쓰나:** 주장이 관문 코앞에서 막혔을 때 — 반박은 견뎌 냈지만 현장 근거가 없어 `hardened → evergreen` 관문을 못 넘을 때.

**무슨 일이 일어나나:** 그걸 판가름할 실험을 사전 등록합니다: 가설, **실행 전에 못 박아 두는 CONFIRM/REFUTE 기준**(그래야 결과를 사후에 합리화할 수 없습니다), 기록할 조건, 그리고 쉬운 말로 정리한 목표를 외부 러너에 넘깁니다.

**무엇은 안 하나:** 코드는 절대 실행하지 않습니다. 설계와 기록만 하고, 실제 실행은 바깥에서 일어나며, 그 결과는 새 출처가 되어 돌아옵니다.

**그다음엔:** 실험을 바깥에서 돌리고 → 결과를 `reharm:root`로 넣으면 → 다음 `reharm:reharmonization`이 그것을 `## Field Evidence`로 import합니다. 바로 이게 `evergreen` 관문을 여는 열쇠입니다.

**이렇게 말해도 됩니다:** *"실험 설계"*, *"실험 사전등록"*.

### `reharm:ensemble` — 답 쓰기

**언제 쓰나:** 단단한 코어가 갖춰졌고, 스코프의 중심 질문을 한 페이지로 실제로 답하고 싶을 때 — 보고서, 의사결정, 또는 인수인계용으로. 루프의 **출구**입니다.

**무슨 일이 일어나나:** 살아남은 것들(hardened 주장, mashup)을 모아 `deliverables/` 페이지 하나로 엮습니다. 하중을 받는 모든 문장이 `(status · confidence · generation)` 스냅샷과 함께 노드를 인용하고, **가장 약한 하중 주장이 페이지의 신뢰도 하한**을 정하며, 미해결 유의점은 `## Open caveats`에 그대로 남습니다.

**무엇은 안 하나:** 어떤 노드의 상태도 바꾸지 않습니다 — 답이 위키를 따라가지, 그 반대가 아닙니다. 다시 돌리면 같은 페이지가 제자리에서 재도출되고, 쓰는 건 `deliverables/`와 마무리 파일(`index`·`hot`·`log`)뿐입니다.

**그다음엔:** 코어를 계속 진화시키고, 나중에 ensemble을 다시 돌리면 페이지가 스스로 갱신됩니다.

**이렇게 말해도 됩니다:** *"답변 합성"*, *"결론 합성"*, *"앙상블"*.

### `reharm:loop-setup` — 자율 루프 설정하고 시작하기

**언제 쓰나:** 위의 일곱 스킬을 *무인으로*(야간 배치, 장기 누적) 돌리고 싶은데 루프 템플릿을 손으로 고치기는 싫을 때.

**무슨 일이 일어나나:** 스코프를 탐지하고, 진짜 결정만 인터뷰로 묻고(`MAX_ITERS`, `RUN_EXPERIMENTS`, 형제 스코프, dynamic/interval 페이싱), 기계적인 필드는 알아서 채우고, 실험 게이트를 첫 tick *전에* 검증하고, `.claude/loop.md`를 쓴 뒤, 같은 호출 안에서 네이티브 `/loop`을 시작합니다.

**무엇은 안 하나:** 지식에 대해서는 아무것도 결정하지 않습니다. 무인 자동 결정은 이 위자드가 쓰는 루프 템플릿에 삽니다([자율 루프 가이드](../templates/loop.guide.ko.md) 참고) — 그리고 그건 당신이 인터뷰에 답하고 루프가 시작된 뒤에야 발동합니다.

**그다음엔:** 세션을 열어 두고 머신을 깨워 두세요. `Esc`로 멈춥니다. 플러그인을 업그레이드했으면 위자드를 다시 돌려 템플릿 수정을 반영하세요.

**이렇게 말해도 됩니다:** *"자율 루프 설정"*, *"루프 시작"*, *"reharm loop setup"*.

## 자연스러운 순서 (단, 일직선은 아닙니다)

스코프는 대개 이렇게 흐릅니다:

```
root  ──▶  reharmonization  ──▶  critique  ──▶  experiment-design  ──▶  ensemble
(씨앗)     (단단하게)            (판정)          (실세계 증명)          (답)
                    ╲
                     ╲──▶  modal-interchange   (두 번째 스코프가 생기면 언제든)
```

하지만 앞에서 뒤로 곧이곧대로 걷는 일은 드뭅니다. 새 자료는 언제든 `root`로 다시 들어오고, `critique`는 찌꺼기가 쌓일 때마다 돌며, `modal-interchange`는 스코프가 둘이 되어야 의미가 생기고, `ensemble`은 잠정 답을 얻으려 일찍 돌린 뒤 코어가 단단해질수록 다시 돌릴 수 있습니다. 지금 어느 단계인지 헷갈리면 `reharm:pushing`이 상태를 읽고 알려 줍니다 — 그게 바로 이 스킬의 존재 이유입니다.

## 기억해 둘 두 가지 약속

- **세대는 획득하는 것이지 주어지는 게 아닙니다.** 세대를 올리는 건 오직 `reharm:reharmonization`뿐이고, 그것도 반박을 견뎌 낸 노드에만 해당합니다. `critique`도 `pushing`도 세대를 올리지 않습니다.
- **파괴되는 것은 없습니다.** 붕괴한 주장은 삭제가 아니라 `deprecated`로 남고, mashup의 원본은 복사가 아니라 인용되며, deliverable은 인용하는 노드를 절대 수정하지 않습니다. 언제든 거슬러 올라갈 수 있습니다.
